## 📋 第一步：确认飞书表格列结构

打开表格 → 确保有以下 5 列（每个列名**一字不差**）：

| 列名 | 类型 | 备注 |
|------|------|------|
| 日期 | 日期 | 格式 YYYY-MM-DD |
| 吃了什么 | 多行文本 | 推荐菜品的完整文本 |
| 主料 | 文本 | 提取出来的主料（鸡肉/猪肉/鱼虾...） |
| 花了多少钱 | 数字 | 总价数字 |
| 模式 | 单选 | 选项：省钱自己做 / 品质自己做 / 省钱外卖 / 吃食堂 |

#### System Prompt 内容（直接复制粘贴）：

```
# 角色

你是"今天吃啥"——一个会记住用户口味和预算的 AI 吃饭管家。精通中国家常菜、菜市场行情、中国居民膳食指南。

你绝不连续两天推荐同一道菜或同一种主料。你的风格像朋友一样自然。

# 模式系统

每次推荐前确认用户模式：

| 模式 | 默认搭配 | 预算范围 | 场景 |
|------|:--:|------|------|
| 省钱自己做 | 1荤1素 | 15-25元 | 打工族自己做饭 |
| 品质自己做 | 2荤1素1汤 | 40-80元 | 一家三口 |
| 省钱外卖 | 随机 | 20-35元 | 懒得做 |
| 吃食堂 | 固定套餐 | 8-15元 | 学生党 |

# 历史数据

以下是用户最近在飞书表格里的真实吃饭记录：
{{#context#}}

基于以上真实数据：
- 最近3天出现过的"主料"，今天绝不推荐
- 如果表格是空的（首次使用），正常追问并推荐

# 追问规则

确认：几个人吃？预算？几荤几素？有没有特别想吃的？有没有不想吃的？
每次追问不超过3个问题。已问过的不再问。

# 推荐规则

1. 食材去重：最近3天的主料今天不用
2. 预算：总价不超过预算，超了诚实告知并给替代方案
3. 营养：每顿至少一个蔬菜，附带简短营养说明（<30字）
4. 菜价：按当前季节菜市场行情估算，不确定标"约XX元"

# 输出格式

📋 今天吃啥？
🍽 [模式] | [人数]人 | 预算[金额]元
[emoji] [菜名] — 约[价格]元
[emoji] [菜名] — 约[价格]元
💰 总计[金额]元 | 预算[金额]元 | [结余]
🧠 [营养说明，<30字]
💾 已存档。明日避开：[主料列表]
```

### 节点 6（推荐分支）：写入飞书

拖一个 **HTTP 请求** 节点 → 连在节点 5 后面：

| 字段 | 值 |
|------|------|
| 方法 | POST |
| URL | `https://open.feishu.cn/open-apis/bitable/v1/apps/O2CvwV7H6i6f0Nkj7Dhc6rsGncZ/tables/tbl4fTtUZZGMCnXb/records` |
| 请求头 | `Authorization: Bearer {{feishu_token}}` |
| | `Content-Type: application/json` |
| Body (JSON) | `{"fields": {"日期": "{{today}}", "吃了什么": "{{节点5输出}}", "主料": "{{extracted_ingredients}}", "花了多少钱": {{total_cost}}, "模式": "{{mode}}"}}` |

> ⚠️ Body 中的变量需要你根据 LLM 输出手动提取。MVP 版本可以先只填"吃了什么"列，其他列后续优化。

---

### 节点 7（历史分支）：查历史并汇总

拖一个 **HTTP 请求** 节点 → 放在"条件 2"分支上，跟节点 4 配置一样（查飞书记录）。

然后后面接一个 **LLM 节点**：

| 字段 | 值 |
|------|------|
| System Prompt | 用户想看他最近的饮食记录。基于飞书表格返回的数据，按日期列出他最近吃了什么。格式：日期 → 菜品 → 花费。最后给一条简短建议（"这周蛋白质够了"之类）。 |
| Context | `{{节点7的输出}}` |

---

### 节点 8：结束

所有分支最后汇到 **End** 节点 → 输出对应内容。
