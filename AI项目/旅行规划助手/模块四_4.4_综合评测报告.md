## 二、评测体系总览

```
                     ┌──────────────────┐
                     │   AI 评测三层体系   │
                     ├──────────────────┤
                     │ ① 自动评测        │ ← 格式合规 / 关键词 / 安全红线
                     │ ② 人工评测 ⭐     │ ← 五维评分 + 测试用例（本次重心）
                     │ ③ 线上评测        │ ← 漏斗 + 成本 + 采纳率（北极星）
                     └──────────────────┘
```

### 五维评分模型

| 维度 | 权重 | 1分（不可用） | 3分（可用） | 5分（优秀） |
|------|:--:|------|------|------|
| 准确性 | 30% | 事实错误、路线不通 | 基本正确，有小瑕疵 | 信息精准 + 季节/交通/时间全考虑 |
| 个性化 | 25% | 无视用户画像 | 部分匹配 | 深度贴合：预算×节奏×兴趣全匹配 |
| 可执行性 | 25% | 行程矛盾、时间排不下 | 基本可执行 | 有备选方案 + 缓冲时间 + 天气考虑 |
| 完整性 | 10% | 缺 ≥2 个板块 | 四板块齐但简略 | 四板块详细 + 贴心信息 |
| 语言质量 | 10% | 不通顺、格式乱 | 通顺平淡 | 有温度、语气匹配画像 |

```
综合分 = 准确性×0.3 + 个性化×0.25 + 可执行性×0.25 + 完整性×0.1 + 语言质量×0.1
≥4.0 🟢 | 2.5-3.9 🟡 | <2.5 🔴
```

### 权重设计逻辑

> 旅行产品的核心价值 = 信息准确 + 行程能落地。
> 所以准确性（30%）+ 可执行性（25%）占了总分的大半。
> 个性化 25% 不是拍脑袋——它是我们跟豆包/Kimi 的差异化壁垒。

## 五、数据指标看板

### 北极星

> **行程采纳率** —— 用户是否真的按规划出发了。
> 追踪：对话结尾主动询问 + 7 天内复访 = 采纳信号。

### 成本模型（实测）

| 指标 | 数值 |
|------|------|
| 单次对话 Token | 1,655 |
| 单次成本（DeepSeek V4） | ¥0.00176 |
| System Prompt 占比 | ~90%（1500 tokens） |
| 日活 1000 / 月成本 | ¥158 |
| 日活 10000 / 月成本 | ¥1,584 |
| 降本方案 | 砍 Prompt 冗余 → 省 28% |

### 用户漏斗

```
100% → 进入对话
 70% → 追问确认画像    🔴 优化重点（TC-004 已修）
 55% → 生成完整行程    🟡
 20% → 看完并反馈
  8% → 采纳行程         ⭐ 北极星
  3% → 复访
```

## 七、面试五分钟讲稿

> 以下按面试节奏设计，每段约 1 分钟，共 5 分钟。

### 第 1 分钟：产品是什么
"我搭建了一个 AI 旅行规划 Agent，基于 DeepSeek V4 和 Dify，配了苏州本地 RAG 知识库。核心差异是——它不像通用 AI 直接甩景点清单，而是先识别你是学生、白领还是老人，再匹配穷游、深度还是慢游模式，输出一份能真正落地的专属行程。"

### 第 2 分钟：我怎么评测它
"AI 产品不能靠感觉判断好不好。我建立了一套五维评测体系——准确性、个性化、可执行性、完整性、语言质量——每个维度有 1/3/5 分的明确判定标准。权重不是拍脑袋的：旅行产品核心是信息准 + 能落地，所以准确性和可执行性占了 55%。"

### 第 3 分钟：测试用例和 Bad Case
"我设计了 5 条测试用例，覆盖三个核心画像的边界和异常场景。比如 TC-005 是用户说'100 块玩 3 天苏州'——专门测模型的诚实度。我记录了 5 个 Bad Case，从'追问像填表'到'知识库边界导致卡死'，每个都有根因分析和修复方案。"

### 第 4 分钟：数据指标
"我把北极星定在行程采纳率而不是生成数量——AI 产品不能只看量。我还算了成本模型：实测单次对话 1655 tokens，约 0.0018 元。量化之后发现 System Prompt 占了 90% 成本，砍掉冗余规则后能降 28% 而不影响评测分。"

### 第 5 分钟：我学到了什么
"做这个项目让我意识到 AI PM 不是 Prompt Engineer。你得设计评测体系验证质量、建数据看板追踪效果、排查全链路问题——用户看到'加载中转圈'，根因可能在 Embedding API 限流而不是模型。这种全链路排查能力是 AI PM 和传统 PM 最大的区别。"
