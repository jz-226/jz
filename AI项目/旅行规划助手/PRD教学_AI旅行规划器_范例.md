## 一个好 PRD 长什么样？—— 8 个模块完整演示

---

### 模块 1：背景与目标

**写这个的目的**：让所有人知道「我们为什么要做」和「做完怎么算赢」。

```
【背景】
用户在做旅行规划时，目前的解决方案是：
- 自己搜小红书/马蜂窝 → 信息碎片化，需要手动拼行程
- 找旅行社定制 → 贵（人均 3000+），且行程模板化不灵活
- 用 ChatGPT 问 → 不懂苏州本地信息（景点距离、实时门票）

三者的共同痛点：花时间、不够个性化、做不到「一站式」。

【目标】
- 上线后 1 个月内，DAU 达到 100（验证场景成立）
- 用户完成一次规划的平均时间 ≤ 3 分钟
- 用户对行程的满意率 ≥ 70%（行程结束后问卷打分）
- 模型成本 / 次规划 ≤ ¥0.15
```

**新手常犯错误**：目标写「提升用户体验」——太虚了，没法验证。目标必须带数字。

### 模块 2：用户与场景

**写这个的目的**：统一所有人对「用户长什么样」的认知。

```
【核心用户】

用户 A：大学生 · 穷游打卡型
- 画像：大二女生，月生活费 2000，想周末去苏州玩
- 痛点：预算有限，想去「最出片」的景点，需要公交/地铁路线
- 使用场景：周五晚上规划周六行程，30 分钟内搞定
- 核心诉求：便宜、顺路不绕、拍照好看

用户 B：上班族 · 周末深度型
- 画像：28 岁程序员，单休，想用一天体验苏州文化
- 痛点：时间紧，不想去商业化景点，想吃本地人吃的馆子
- 使用场景：周六早上起床才决定出门，需要快速生成路线
- 核心诉求：避开人挤人、小众体验、好吃大过好看

用户 C：退休老人 · 慢游辅助型
- 画像：65 岁退休教师，女儿帮他用 App，想在苏州玩一周
- 痛点：体力有限，需要知道厕所位置、休息点、轮椅通道
- 使用场景：提前一周规划，女儿打印出来给他带着
- 核心诉求：节奏慢、有休息点、无障碍信息

【使用场景总结】
┌──────────────┬──────────────┬──────────────┐
│  穷游打卡     │  周末深度     │  慢游辅助     │
│  预算敏感     │  时间敏感     │  体力敏感     │
│  < ¥300/天   │  < 8小时/天   │  < 3景点/天   │
└──────────────┴──────────────┴──────────────┘
```

**新手常犯错误**：用户画像写成「所有想旅游的人」→ 太宽了，等于没写。

### 模块 3：功能范围

**写这个的目的**：画边界。做哪些、**不做哪些**，两个都一样重要。

```
【V1.0 功能范围】

✅ 做：
- 3 类用户画像 × 3 种场景的行程生成
- 输入：目的地 + 天数 + 预算 + 兴趣标签
- 输出：完整的日行程（景点顺序 + 交通 + 餐饮 + 预算估算）
- 支持重新生成不满意的一天（单独改某天，不动其他天）
- 行程一键复制/分享

❌ 不做（V1.1 或以后）：
- 酒店/机票预订（不接 API，只规划不交易）
- 实时天气提醒
- 语音交互
- 社交功能（发布行程/点赞）
- 景点 VR 预览
- 多人协作规划
```

**新手常犯错误**：只写「做什么」，不写「不做什么」。开发会默认你觉得「酒店预订」也要做。

### 模块 4：核心流程

**写这个的目的**：用户从哪进来，走几步完成任务？让所有人都对流程有同一张地图。

```
【主流程：从打开到拿到行程】

用户打开 Bot
    │
    ▼
选择模式 ──→ 穷游打卡 / 周末深度 / 慢游辅助
    │
    ▼
输入信息 ──→ 目的地 / 天数 / 预算 / 兴趣标签
    │
    ▼
Bot 生成行程 ──→ 展示日行程（景点→交通→餐饮→预算）
    │
    ├── 满意 ──→ 复制/分享 → 结束
    │
    └── 不满意 ──→ 点击「重新生成第 X 天」
                        │
                        ▼
                  Bot 只重排那一天，其他天不变 → 回到展示

【异常流程】（新手最容易漏！）

1. 用户输入模糊 → Bot 追问确认
   例：「想去苏州」→ 「苏州哪个区？姑苏区/工业园区/吴中区？」
   
2. 预算过低 → Bot 提示并建议调整
   例：「预算 50 元/天」→ 「这个预算可能只够交通和简餐，建议调整为至少 150 元」

3. 网络超时 → 提示用户稍后重试，保留已输入的信息

4. 景点关闭 → 自动替换为同类型备选景点
   例：周一博物馆闭馆 → 替换为平江路步行街

5. Bot 生成内容不合规 → 走安全审核拦截，返回通用提示
```

**新手常犯错误**：只画正常流程。上线后 90% 的 Bug 都出在异常流程上。

### 模块 5：功能详情

**写这个的目的**：这是 PRD 最厚的部分。每个功能怎么交互、边界条件是什么、数据怎么处理。

```
【功能 F1：模式选择】

入口：用户进入 Bot 后，看到的第一条消息

交互：
- Bot 发送欢迎语 + 3 个选项按钮
  「Hi！我是你的旅行规划助手 👋
   你想怎么玩？
   [穷游打卡] [周末深度] [慢游辅助]」

- 用户点击按钮 → 进入模式
- 用户如果直接打字不点按钮 → Bot 根据内容自动判断模式
  例：「我想省钱」→ 匹配穷游打卡

边界条件：
- 如果用户输入跟 3 种模式都不匹配 → 追问「你是更在意预算、时间、还是舒适度？」
- 如果用户切换模式 → 清空之前输入的行程信息，重新开始


【功能 F2：行程生成】

入口：用户完成模式选择 + 信息输入后

输入参数：
| 参数 | 必填 | 格式 | 示例 |
|------|------|------|------|
| 目的地 | 是 | 城市+区域 | 苏州·姑苏区 |
| 天数 | 是 | 整数 1-7 | 2 |
| 预算 | 是 | 整数（元） | 500 |
| 兴趣标签 | 否 | 多选 | 园林/美食/拍照/历史/亲子 |

生成逻辑（给开发看的）：
1. 从知识库检索符合「目的地+兴趣标签」的景点列表
2. 按「预算」过滤（门票累加 ≤ 日均预算）
3. 按「天数」分配景点，每天 ≤ 3 个（考虑距离顺路）
4. 每天包含：上午景点 → 午饭推荐 → 下午景点 → 晚饭推荐 → 交通说明
5. 输出总预算 = 门票 + 餐饮估算 + 交通估算

异常处理：
- 知识库中无匹配景点 → 「抱歉，目前还没有收录 XX 区的信息，我可以用苏州经典路线替代吗？」
- 预算不足以覆盖任何景点 → 提示调整
- 天数为 1 但输入 5 个兴趣标签 → 自动按热度截取 Top 3

输出格式：
```
📋 你的苏州 2 日行程

【Day 1 — 预算约 ¥280】
🌅 上午：拙政园（门票 ¥80）
  → 建议游览 2 小时
🍜 午饭：吴门人家（人均 ¥60）
  → 推荐：松鼠桂鱼、响油鳝糊
🌤 下午：苏州博物馆（免费，需预约）
  → 贝聿铭设计，建筑本身值得看
🍲 晚饭：同得兴面馆（人均 ¥30）
  → 枫镇大面是招牌
🚇 交通：地铁 1 号线 → 4 号线，约 ¥8

【Day 2 — 预算约 ¥180】
...

💰 总预算：¥460（门票 ¥80 + 餐饮 ¥180 + 交通 ¥16 + 其他 ¥184）
📝 提示：苏州博物馆周一闭馆，建议避开周一出行
```
```

### 模块 6：数据与埋点

**写这个的目的**：上线后你怎么知道产品做对了？看什么数据？

```
【核心指标】
| 指标 | 定义 | 目标值 | 为什么看它 |
|------|------|--------|----------|
| DAU | 日活跃用户数 | ≥ 100 | 验证场景需求存在 |
| 完成率 | 生成行程成功 / 开始规划 | ≥ 80% | 流程是否顺畅 |
| 平均生成时间 | 从输入到拿到行程 | ≤ 3 分钟 | 体验效率 |
| 重新生成率 | 点击「重新生成」的比例 | ≤ 30% | 首次生成质量 |
| 满意度 | 行程结束后问卷打分 | ≥ 4.0/5 | 行程质量 |
| 单次成本 | Token 消耗 × 单价 | ≤ ¥0.15 | 模型成本可控 |

【需要埋的点】
1. 用户打开 Bot（DAU）
2. 选择模式（哪个模式最受欢迎？）
3. 输入信息后点击生成（开始规划）
4. 生成成功 / 失败（完成率 + 错误率）
5. 点击「重新生成」（不满意率）
6. 点击「复制/分享」（用户真正用了行程）
```

### 模块 7：验收标准

**写这个的目的**：什么状态算「可以上线」？QA 知道测什么，你知道什么叫「过了」。

```
【上线前必须通过】

功能完整性：
☐ 3 种模式都能正常进入和切换
☐ 输入不同的目的地/天数/预算/兴趣标签，都能生成结果
☐ 重新生成单天功能正常
☐ 复制/分享功能正常

异常处理：
☐ 输入为空 → 有提示
☐ 输入预算 1 元 → 合理提示
☐ 输入目的地「火星」→ 友好提示不支持
☐ 输入天数 30 天 → 提示上限 7 天
☐ 网络断开时 → 不崩溃，有重试提示

内容质量（抽检）：
☐ 随机抽 10 个行程，所有景点真实存在
☐ 门票价格误差 ≤ 20%
☐ 餐饮推荐在地图上 2 公里内有结果
☐ 交通方式合理（不会让人从姑苏区打车到昆山吃午饭）

性能：
☐ 生成一个 2 天行程 ≤ 30 秒
☐ 重新生成单天 ≤ 15 秒

安全：
☐ 不输出违法/色情/政治敏感内容
☐ 不泄露系统 Prompt
```

### 模块 8：风险与依赖

```
【依赖】
| 依赖方 | 依赖什么 | 如果掉链子怎么办 |
|--------|---------|----------------|
| DeepSeek API | 模型推理能力 | 备选：Kimi API，Prompt 需微调 |
| 知识库（苏州景点） | 景点信息准确 | 上线前人工抽检 50 条景点数据 |
| Dify 平台 | Bot 运行基础设施 | Dify 挂掉则 Bot 不可用 |

【风险】
| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 模型幻觉编造假景点 | 中 | 高 | 知识库约束 + 输出中标注「信息来自知识库」 |
| Token 成本超预算 | 中 | 中 | 加成本监控告警，超 ¥0.2/次 自动限流 |
| 高峰期 API 超时 | 低 | 高 | 设置 30 秒超时，超时后返回友好提示 |
| 用户输入不当内容 | 低 | 中 | 输入层加敏感词过滤 |
```
