## 一、三个概念一句话

| 概念 | 口诀 |
|------|------|
| **Chatbot** | 问一句答一句，没记忆没工具 |
| **Workflow** | 按固定路线走，每一步都画得出来 |
| **Agent** | 自己判断下一步，能查状态、能调工具、能记住 |

**判断标准**：能不能把所有路径画出来？能 → Workflow。不能 → Agent。

---

## 二、Agent 循环（4 个字口诀）

```
察 → 想 → 动 → 回
```

- **察**：现在什么情况？（用户说了啥、数据库里有什么）
- **想**：我该干什么？（生成方案？追问？查打卡？）
- **动**：执行（调 LLM / 写数据库 / 发提醒）
- **回**：看结果，决定下一步

Chatbot 没有循环。它只跑一次：察 → 想 → 答。结束。

## 三、Context Engineering vs Prompt Engineering

| | Prompt Engineering | Context Engineering |
|------|------|------|
| 管什么 | 模型说啥 | 模型看到啥、能用啥、记住啥 |
| 一句话 | "写指令" | "造世界" |

**Context Engineering 包含**：对话历史管理、RAG 检索策略、工具设计与权限、记忆存储、安全护栏、评测体系。

**问自己的唯一问题**：这个节点的输出拼进 Prompt 之后，模型读到的是有用信息还是垃圾？

## 五、Harness 六层（口诀：工记权运反评）

| 层 | 干什么 | 没有会怎样 |
|------|------|------|
| **工** 具层 | 让 Agent 能干 | 只会聊天屁事做不了 |
| **记** 忆层 | 让 Agent 能记住 | 每次都是第一次见你 |
| **权** 限层 | 不让 Agent 乱干 | 用户让删库模型真去调 API |
| **运** 行环境 | 给大脑配身体 | 光有 Prompt 什么都调不了 |
| **反** 馈层 | Action 后看结果 | 调完 API 不知道成功没 |
| **评** 测层 | 知道好还是差 | 改 Prompt 全凭感觉 |

## 六、面试必背三句话

**1. "为什么用 Workflow 不用 Agent？"**

> 旅行规划器的路径是确定的——追问画像→确认模式→生成行程→用户调整，每一步都能画出来。所以它用 Chatflow 就够了，不需要 Agent。

**2. "为什么儿童康复助手要升级成 Agent？"**

> 因为加了数据库之后，行为取决于运行时数据——打卡率高鼓励、打卡率低提醒康复师、新用户生成方案、老用户显示今日计划。这些路径画不完，Agent 需要自己判断下一步。

**3. "Context Engineering 和 Prompt Engineering 有什么区别？"**

> Prompt Engineering 是写指令——管模型说什么。Context Engineering 是造世界——管模型能看到什么、用什么工具、记住什么、有什么权限。我排查过 Embedding 限流导致的卡顿、知识库空返回导致的卡死、追问 5 连击导致的用户流失——每一个翻车都是 Context Engineering 的问题，不是 Prompt 问题。

---

## 七、核心原则（背这句就够了）

> **能用 Workflow 解决的不要用 Agent。Agent 不是更高级，是适用场景不同。**
