# Stage 0：理解 Agent 是什么

> 姜梓 · 2026.07.08
> 目标：搞懂 Agent vs Workflow vs Chatbot，搞懂什么时候不该用 Agent

---

## 三个核心概念一句话分清

| 概念 | 一句话 | 我的项目里谁是谁 |
|------|------|------|
| **Chatbot** | 问一句答一句，没记忆没工具 | 最早的旅行规划器——输入"苏州两天"→ 输出行程，结束 |
| **Workflow** | 按固定步骤走，每步输出是下步输入 | Dify Chatflow——先追问画像 → 再生成行程 → 再拼打卡表 |
| **Agent** | 自己决定下一步做什么，能调用工具，能记住状态 | 儿童康复助手 Agent 版——生成方案 → 写入数据库 → 检查打卡率 → 低于阈值自动提醒 |

---

## Agent 基本循环

```
      ┌──────────────────────────┐
      │                          │
      ▼                          │
  [Observe]  ← 读取环境：用户说了什么、数据库里有什么、上次做了什么
      │
      ▼
  [Think]    ← 模型推理：我现在该生成方案？还是该追问他年龄？还是该查打卡数据？
      │
      ▼
  [Act]      ← 执行：调用 LLM 生成方案 / 调 HTTP 写数据库 / 返回追问
      │
      └────────→ 回到 Observe，根据结果决定下一步
```

**Chatbot 没有这个循环。它只跑一次：Observe → Think → 回答。然后就没了。**

我的儿童康复助手升级成 Agent 之后才真正有了这个循环——每次打卡写库 → 下次打开读取历史 → 根据完成率决定要不要提醒。

---

## Workflow 和 Agent 的边界（什么时候不该用 Agent）

| 场景 | 该用什么 | 为什么 |
|------|------|------|
| 用户输入→固定处理→输出，路径确定 | **Workflow** | 不需要自主决策，流程越确定越可靠 |
| 用户输入不完整，需要追问 | **Workflow 加条件分支** | 追问是固定逻辑，不是自主决策 |
| 需要记住多轮对话状态 | **Workflow + 变量节点** | 变量存状态就够了，不需要 Agent |
| 需要根据运行时数据动态决定下一步 | **Agent** | 比如"打卡率 < 50% → 通知康复师，> 80% → 鼓励家长" |
| 需要同时操作多个工具且顺序可变 | **Agent** | 比如先查数据库 → 再调 LLM → 再发通知 |

**关键判断**：如果你能画出所有可能的路径 → Workflow 就够了。如果路径太多画不全 → 才需要 Agent。

---

## Agent 失败的常见原因

| 原因 | 表现 | 我踩过的坑 |
|------|------|------|
| **上下文污染** | 对话太长后模型"忘"了最初要干嘛 | 旅行规划器问多轮后开始推荐不相关的景点 |
| **工具设计差** | Agent 不知道什么时候该调哪个工具 | — |
| **状态不可恢复** | 中间一步挂了，整个流程崩了，没法从断点继续 | Dify 知识库空返回导致整个管道卡死（连云港问题） |
| **没有评测** | 不知道 Agent 到底做对了还是做错了 | 做五维评分之前我就是瞎改 Prompt |

## 回答三个问题

### 问题 1：我的场景为什么需要 Agent，而不是普通 Workflow？

**旅行规划器 → Workflow 就够了。**

用户输入 → 追问画像 → 生成行程 → 用户调整 → 重新生成。这条路径是**确定性的**——每一步都画得出来，不需要模型自己决定"我现在该干嘛"。所以它是个 Chatflow，不是 Agent。

**儿童康复助手 → 需要 Agent。**

因为加了数据库之后，路径变了：

- 新用户 → 生成方案 → 写库 → 返回方案
- 老用户 → 读库 → 检查打卡率 → 如果太低 → 提醒康复师 → 如果正常 → 显示今日训练
- 家长打卡 → 写库 → 检查是否连续断 2 天 → 是 → 推送提醒

这些路径**取决于运行时数据**，没法全部画死在 Workflow 里。Agent 自己判断下一步该做什么。

**关键判断**：如果产品只需要"输入→输出"（哪怕中间有追问有 RAG），Workflow 够用。如果产品需要"记住状态→根据状态决定动作→影响下次状态"，才需要 Agent。

### 问题 2：Context Engineering 和 Prompt Engineering 的区别是什么？

| | Prompt Engineering | Context Engineering |
|------|------|------|
| **管什么** | 管"模型被问的时候看到什么" | 管"模型活在一个什么样的环境里" |
| **范围** | System Prompt + User Message | Prompt + 工具定义 + 记忆存储 + RAG 检索结果 + 对话历史窗口 + 权限边界 |
| **举例** | 写给康复助手的那段 4 层 Prompt | 整个 Dify 编排：知识库怎么检索、变量怎么存、HTTP 节点怎么写数据库、对话历史保留多少轮 |
| **谁操心** | PM + 算法 | PM + 工程 + 架构 |

**一句话**：Prompt Engineering 是写指令。Context Engineering 是造世界——这个世界里模型能看到什么、能用什么工具、能记住什么、有什么权限。

**Context Engineering 包含但不限于**：
- 对话历史管理（保留最近几轮？怎么做摘要？）
- RAG 检索策略（Top-K 多少？什么时候触发检索？检索为空怎么办？）
- 工具设计与权限（Agent 能调哪些 API？什么条件下不能调？）
- 记忆管理（哪些状态存变量？哪些存数据库？什么时候清？）
- 安全护栏（危险动作拦截规则、医疗免责声明）

### 问题 3：Harness 的各层分别解决什么问题？

Harness = 模型外面的那层壳。模型只负责推理，Harness 提供一切推理所需的东西。

```
┌─────────────────────────────────────────┐
│              Harness                     │
│                                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌────────┐  │
│  │ 工具层 │ │ 记忆层 │ │ 权限层 │ │ 运行环境 │  │
│  └──────┘ └──────┘ └──────┘ └────────┘  │
│                                          │
│  ┌──────┐ ┌──────────────────────────┐  │
│  │ 反馈层 │ │       评测层              │  │
│  └──────┘ └──────────────────────────┘  │
│                                          │
│        ┌─────────────┐                   │
│        │   模型（推理）  │                   │
│        └─────────────┘                   │
└─────────────────────────────────────────┘
```

| 层 | 解决什么问题 | 没有这层会怎样 | 我的项目里对应的东西 |
|------|------|------|------|
| **工具层** | 模型只能说话，需要工具才能干活（查数据库、发通知、写文件） | Agent 变成只会聊天的废人 | Dify HTTP 节点写云开发数据库 |
| **记忆层** | 模型本身无状态，必须外部存储对话历史和业务数据 | 每一轮对话都是"第一次见到你" | 变量节点存孩子名+训练计划；云数据库存打卡记录 |
| **权限层** | 限制 Agent 能做什么、不能做什么 | 用户问一句"把数据库删了"模型可能真去调 API | 危险动作拦截规则（倒立→拒绝）、医疗免责 |
| **运行环境** | 模型只是脑子，需要一具身体来跑代码、调 API、管理状态机 | 光有 Prompt 什么都干不了 | Dify Workflow 编排引擎 |
| **反馈层** | Agent 做完一个动作后需要知道结果，才能决定下一步 | Agent 调完 API 不知道成功没，盲目继续 | 打卡写入后返回成功/失败 → 决定是否重试 |
| **评测层** | 不知道 Agent 做得好不好就是瞎迭代 | 改了 Prompt 感觉"好像好了点"但没法量化 | 五维评分 + 5 条测试用例 + Bad Case 追踪 |

---

## 总结：写进面试的一句话

> "我做了三个项目，从 Chatbot 做到 Workflow 做到 Agent 雏形。旅行规划助手是 Workflow——路径确定，追问→生成→调整。儿童康复助手我开始加 Agent 能力——因为它需要记住每个孩子的训练状态，根据打卡率动态决定下一步动作，这不是 Workflow 画得完的。整个过程中我越来越清楚一个原则：**能用 Workflow 解决的不要用 Agent，Agent 不是更高级，是适用场景不同。**"
